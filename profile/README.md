# RDB 중심 스터디 witih Spring

## 개요

- **학습 목표**: RDB를 최대한 활용하며 정합성과 성능을 고려한 백엔드 엔지니어링 역량 향상
- **학습 사이클**: 이론 → 실습 → 분석 → 회고
- **학습 단계**: 이론과 실습(쇼핑몰 프로젝트)을 병렬 진행하며 단순한 구조에서 복잡한 구조로 확장
- **학습 방향**: RDB 중심으로 학습하며 Spring 프레임워크는 프로젝트에 적용할때마다 핵심 학습
- **진행 방식**: '개념' 설명 1명 + 각자 '이론 → 실습 → 분석 → 회고' 과정 설명 → 질의

## 전체 로드맵

### 방향

* **개인 프로젝트**: 스스로 문제를 재현하고 해결해나가며, RDB를 최대한 활용해서 문제를 해결하는 역량 향상 (문제 자체를 문제로 인식하고 정의하는 역량 강화)
* **팀 프로젝트**: 문제 발생을 최소화 하는 RDB 설계  (시니어 소프트웨어 엔진니어 RDB 역량)

### 정리

 * 완성도 높은 쇼핑몰 MVP 프로젝트
 * 단, 쇼핑몰이란 주제는 문제를 잘 찾기 위한 '리소스'이다.
 * 문제를 정의하고 분석하는 역량 향상 → 문제를 최소화하는 설계 역량 향상

### 진도

* **1 - 9 주차**: RDB 중심으로 학습하며, 개인 학습용 프로젝트로 소형 커머스 단계적 개선 (기본 강의안 스키마에서 확장)
* **10 - 12 주차**: RDB 중심으로 학습하며, 팀 프로젝트로 소형 커머스 개발 (실서비스 스카마 설계)

| 주차  | 이론 주제                                                                  | 실습 예시 (쇼핑몰)                                                                      |   강의안 범위   | 추가 스터디             |
| :-: | :--------------------------------------------------------------------- | :------------------------------------------------------------------------------- | :--------: | ------------------ |
|  1  | (1) 데이터 관리, 조회와 정렬<br>(2) 데이터 가공, 집계와 그룹핑                              | JDBC CRUD                                                                        |  입문편 4~7장  | Spring DB 접근 핵심    |

|  4  | 인덱스 1 (B-Tree, EXPLAIN, 단일 인덱스)                                        | 대량 데이터 넣고 EXPLAIN 실전                                                             |   기본편 8장   |                    |
|  5  | 인덱스 2 (복합, 커버링, 설계)                                                    | 쇼핑몰 조회 성능 튜닝                                                                     |   기본편 9장   |                    |

|  7  | 트랜잭션 2 - 격리 수준 (동시성 문제: Dirty Read, Non-Repeatable Read, Phantom Read) | 각 격리 수준에서 어떤 현상이 발생/차단되는지 직접 재현하고 해결 (재고 차감, 주문 총액의 중간에 변경, 할인 적용 대상 집계가 꼬이는 경우) | 기본편 10~11장 |                    |

| 11  | 팀프로젝트: 스키마 발표 & 합의                                                     | 스키마 발표 비교 → 팀 스키마 합의 → 통합 DDL                                                    |     —      | DB 장애 대응 전략 수립     |
| 12  | 팀프로젝트: 최종 완성                                                           | 각 도메인별 API 리팩토링 & 병합 → EC2/RDS 배포, 회고                                            |     —      | 좋은 시스템 디자인이란?      |

### 학습 사이클 (이론 → 실습 → 분석 → 회고)

| 단계  | 목적                                                               |
| :-- | :--------------------------------------------------------------- |
| 이론  | 개념 학습                                                            |
| 실습  | 이론을 적용할 실습 문제를 **스스로 설계**하고 수행 (문제 상황 재현 → 적용해서 해결)              |
| 분석  | 실습 결과가 왜 그렇게 나왔는지 원리 이해/원인 추적 (`EXPLAIN`, 로그, 실행 시간 근거 제시)       |
| 회고  | 스스로 묻고 답하기 ("다르게 했다면?", "어떤 상황에서 해결책이 적합할지?", "실무에서는 어떻게 할지?" 등) |

**'실습' 단계에서 더미데이터 적재 후 부하테스트 적극 활용 추천**: EC2 (API 테스트 툴) ↔ EC2 (Spring Application) ↔ RDS (MySQL)
* 순수 RDB 비용에 가깝게 계측할 경우, 동일 VPC에 EC2(Client)와 EC2(Server)생성해서 Private IP로 통신 (비용 무료)
* 네트워크 비용도 계측에 포함시켜야될 경우, 로컬(Client)에서 EC2(Server)로 통신 필요 (비용 유료)
* 비용 부담시 로컬 환경에서 수행 가능 (단, OS에 다수의 프로세스 실행으로 계측의 오차가 발생)


### 학습 사이클 참고 예시: [주차별 학습 사이클 예시 바로가기](https://github.com/ureca-RDB-with-Spring-study/.github/wiki)

- **이론**: "`EXPLAIN`의 `type`이 `ALL`이면 풀스캔, `range`면 인덱스 범위 스캔, `ref`면 인덱스 정확 매치다. `ALL`이 나오는 쿼리는 의심부터 해야 한다."
- **실습**: "`customers` 테이블에 100만 건을 넣고, 이메일로 조회하는 쿼리를 돌려보자. 인덱스 전/후의 `EXPLAIN`과 실제 시간을 비교한다."
- **분석**: "인덱스 없을 때 1.2초, 인덱스 추가 후 0.003초. 약 400배 차이. `EXPLAIN`에서 `type`이 `ALL`에서 `ref`로 바뀌고, `rows`도 100만에서 1로 줄었다."
- **회고**: "인덱스는 만능이 아니다. `INSERT / UPDATE` 시엔 인덱스도 같이 갱신돼야 하므로 쓰기 성능은 느려진다. 쓰기와 읽기의 트레이드오프를 실제로 재봐야겠다."

## 스터디 진행 방식

### 학습 후 기록

1. Organization에 각자 REPO 생성 (REPO명: 깃허브아이디-rdb-with-spring)
2. 주제마다 브런치 생성해서 커밋 및 푸시 (ex. branch: index, transaction)
3. 각자 REPO의 WIKI에 주제마다 학습 글 기록 (주차마다 Page 생성해서 WIKI 사이드바에 기재: )
	1. 이론 
	2. 실습 (스키마 포함) 
	3. 분석
	4. 회고
4. 각자 REPO의 README.md에 WIKI에 업로드한 글 링크 달기

### 모임 진행 (매주 수요일 18시, 약 2시간 내외)

1. **개념 설명(20분)**: 1명의 담당자가 해당 주차 개념 설명 및 질의
2. **이론 → 실습 → 분석 → 회고 공유(각 20분)**: 각자가 만든 이론에 관한 실습 문제, 분석 내용과 회고를 발표 및 질의
3. **토론과 정리(20분)**: 더 깊게 파고 싶은 주제나 앞으로 방향성 토른과 정리
