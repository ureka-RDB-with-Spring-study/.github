# RDB 중심 스터디 with Spring

## 개요

- **학습 목표**: RDB를 최대한 활용하며 정합성과 성능을 고려한 백엔드 엔지니어링 역량 향상
- **학습 사이클**: 이론 → 실습 → 분석 → 회고
- **학습 단계**: 이론과 실습(쇼핑몰 프로젝트)을 병렬 진행하며 단순한 구조에서 복잡한 구조로 확장
- **학습 방향**: RDB 중심으로 학습하며 Spring 프레임워크는 프로젝트에 적용할때마다 핵심 학습
- **진행 방식**: '이론' 설명 1명 + 각자 '실습 → 분석 → 회고' 과정 설명 → 질의

## 전체 로드맵

* **1 - 9 주차**: RDB 중심으로 학습하며, 개인 학습용 프로젝트로 소형 커머스 단계적 개선 (기본 강의안 스키마에서 확장)
* **10 - 12 주차**: RDB 중심으로 학습하며, 팀 프로젝트로 소형 커머스 개발 (실서비스 스카마 설계)

| 주차  | 이론 주제                                                             | 실습 (쇼핑몰)                                    |   강의안 범위   | 추가 스터디               |
| :-: | :---------------------------------------------------------------- | :--------------------------------------------- | :--------: | -------------------- |

### 기술 스택

- **DB**: MySQL
- **애플리케이션**: Java, Spring 
- **인프라**: AWS EC2 & RDS (10주차 이후 통합 단계에서 도입)
- **CI/CD**: Github Actions

### 학습 사이클 (이론 → 실습 → 분석 → 회고)

| 단계  | 목적                                                   |
| :-- | :--------------------------------------------------- |
| 이론  | 개념 학습                                                |
| 실습  | 이론을 적용할 실습 문제를 **스스로 설계**하고 수행                       |
| 분석  | 실습 결과가 왜 그렇게 나왔는지 원인을 추적. `EXPLAIN`, 로그, 실행 시간 근거 제시 |
| 회고  | 스스로 묻고 답하기. "다르게 했다면?", "실무에선 어떻게 쓰일까?"              |

#### 주차별 참고 예시: [RDB 중심 스터디 with Spring: 학습 사이클 예시](https://github.com/ureka-RDB-with-Spring-study/.github/wiki)

- **이론**: "`EXPLAIN`의 `type`이 `ALL`이면 풀스캔, `range`면 인덱스 범위 스캔, `ref`면 인덱스 정확 매치다. `ALL`이 나오는 쿼리는 의심부터 해야 한다."
- **실습**: "`customers` 테이블에 100만 건을 넣고, 이메일로 조회하는 쿼리를 돌려보자. 인덱스 전/후의 `EXPLAIN`과 실제 시간을 비교한다."
- **분석**: "인덱스 없을 때 1.2초, 인덱스 추가 후 0.003초. 약 400배 차이. `EXPLAIN`에서 `type`이 `ALL`에서 `ref`로 바뀌고, `rows`도 100만에서 1로 줄었다."
- **회고**: "인덱스는 만능이 아니다. `INSERT / UPDATE` 시엔 인덱스도 같이 갱신돼야 하므로 쓰기 성능은 느려진다. 쓰기와 읽기의 트레이드오프를 실제로 재봐야겠다."

## 진행 방식

### 학습 후 기록

1. Organization에 각자 REPO 생성
2. 주제마다 브런치 생성해서 커밋 및 푸시 (ex. branch: index, transaction)
3. 각자 REPO의 WIKI에 주제마다 학습 글 기록 (주차마다 Page 생성해서 WIKI 사이드바에 기재: 예시 스터디장 레포에 기재)
	1. 이론 
	2. 실습 (스키마 포함) 
	3. 분석
	4. 회고
4. 각자 REPO의 README.md에 WIKI에 업로드한 글 링크 달기

### 모임 진행 (약 2시간 내외)

1. **이론 설명(20분)**: 1명의 담당자가 해당 주차 개념 설명
2. **실습 → 분석 → 회고 공유(각 20분)**: 각자가 만든 실습 문제, 분석 내용과 회고를 발표 (발표중 질의 및 토론)
3. **토론과 정리(20분)**: 더 깊게 파고 싶은 주제나 앞으로 방향성 토른과 정리

### 역할 분담

- **이론 담당**: 매주 랜덤으로 1명 핵심 개념을 팀에 설명
- **실습 → 분석 → 회고**: 전원이 같은 주제를 각자 수행한 과정을 보며 사고 확장
